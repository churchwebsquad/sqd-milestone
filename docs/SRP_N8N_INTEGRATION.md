# SRP Generator — n8n integration

The SRP Generator delegates two long-running jobs to n8n workflows:
**transcription** (sermon video → transcript text) and **clipcutter**
(picked clips → rendered MP4s). This doc covers the n8n side only:
the webhooks the workflows expose, the trigger payloads they
receive, and the callbacks they need to send back.

The Vercel app at `milestone-comms-app` brokers the calls. It doesn't
run any transcription engine or FFMPEG itself — n8n does. Vercel just
fires the webhook, then waits for the callback.

---

## Two webhooks

### Webhook 1 — Transcription

Trigger URL goes in the Vercel env var
`SRP_N8N_TRANSCRIPTION_WEBHOOK_URL`.

**Recommended n8n setup:** Webhook node → "Respond Immediately" mode.
The trigger payload arrives, your workflow does the actual work in
the background, and the workflow's last step POSTs the result to the
`callback_url` from the trigger payload.

### Webhook 2 — Clipcutter

Trigger URL goes in the Vercel env var
`SRP_N8N_CLIPCUTTER_WEBHOOK_URL`. Same "Respond Immediately"
pattern.

---

## Shared secret

A single secret authenticates both directions:

- Vercel includes it in the trigger payload as `callback_secret`
- n8n must echo it back in the callback, either as
  `Authorization: Bearer <secret>` OR as a `callback_secret` field
  in the callback body
- Stored on the Vercel side as the env var `SRP_N8N_CALLBACK_SECRET`

Set the same value on both sides. Any opaque string ≥ 32 chars.

## Transcription contracts

### What Vercel sends to your n8n webhook

```json
{
  "job_id": "<uuid — generated by Vercel per trigger>",
  "session_id": "<text — e.g. 1802_Mosaic_20260608120000>",
  "source_url": "<normalized https URL>",
  "source_type": "youtube | vimeo | dropbox | google_drive | direct | unknown",
  "callback_url": "https://<vercel-host>/api/srp/transcription-callback",
  "callback_secret": "<the shared secret>"
}
```

### What your n8n workflow POSTs back to `callback_url`

```json
{
  "job_id":          "<echo from trigger>",
  "session_id":      "<echo from trigger>",
  "callback_secret": "<the shared secret>",
  "status":          "completed",
  "transcript":      "<the full transcript text — markdown ok>",
  "duration_seconds": 1234,
  "words":           []
}
```

Status values:
- `completed` — Vercel writes `transcript` to the session row
- `failed` — Vercel writes a `__TRANSCRIPTION_FAILED__` sentinel
  + your `error_message` so the UI can show the error
- `processing` — no-op (intermediate progress pings, optional)

### Error payload

```json
{
  "session_id":      "<echo>",
  "callback_secret": "<the shared secret>",
  "status":          "failed",
  "error_message":   "Why the transcription failed (1-2 sentences)"
}
```

## Clipcutter contracts

### What Vercel sends to your n8n webhook

```json
{
  "job_id":      "<uuid>",
  "session_id":  "<text>",
  "source_url":  "<https video URL>",
  "clips": [
    {
      "clip_id":      "1",
      "clip_name":    "Hook on grace",
      "in_point_ms":  145000,
      "out_point_ms": 200000,
      "duration_ms":  55000,
      "quote":        "verbatim quote from transcript",
      "category":     "hook | teaching | application | story | invitation"
    }
  ],
  "creative_direction": null,
  "callback_url":    "https://<vercel-host>/api/srp/clipcutter-callback",
  "callback_secret": "<the shared secret>"
}
```

The `clips[]` array is ordered as the strategist picked them
(Reel 1 = clips[0], Reel 2 = clips[1]). Each `clip_id` is a stable
string — n8n MUST echo it back unchanged in the callback so the
Vercel app can match results to their slots.

### What your n8n workflow POSTs back to `callback_url`

```json
{
  "job_id":          "<echo>",
  "session_id":      "<echo>",
  "callback_secret": "<the shared secret>",
  "status":          "completed | failed | partial",
  "clip_results": [
    {
      "clip_id":   "1",
      "video_url": "https://<wherever the rendered MP4 is hosted>",
      "srt_url":   "https://<optional SRT subtitles>",
      "status":    "done"
    }
  ]
}
```

Per-clip status:
- `done` — `video_url` (and optional `srt_url`) populated
- `failed` — `error_message` populated, no urls

Job-level status:
- `completed` — every clip is `done`
- `partial` — some clips are `done`, others `failed`
- `failed` — the whole job blew up before any clip rendered. If
  `clip_results` is empty when status=failed, Vercel marks every
  still-queued clip as `failed` so the UI doesn't spin forever.

## What if the predecessor app already had n8n workflows?

If `vid2.thesqd.com` already runs SRP workflows for the old version:

1. **Reuse the webhook URLs** — same workflows, same triggers. Just
   make sure the webhook nodes are in "Respond Immediately" mode.
2. **Reuse the callback secret** — set both `SRP_TOOL_N8N_CALLBACK_SECRET`
   (old) and `SRP_N8N_CALLBACK_SECRET` (new) to the same value during
   migration.
3. **Update the callback destination** — the old app likely had n8n
   posting to a hardcoded Supabase Edge Function URL (something like
   `https://...supabase.co/functions/v1/srp-transcript-callback`).
   Replace the hardcoded URL with `{{ $json.callback_url }}` so n8n
   reads the URL from the trigger payload. That makes the workflow
   environment-agnostic — Vercel sends the right callback URL for
   wherever the app is deployed (production vs preview vs local).

## Timing expectations

| Step | Typical time | n8n config |
|---|---|---|
| Vercel → n8n trigger acknowledgment | < 2s | "Respond Immediately" |
| Transcription job | 3–7 min | Async after ack |
| Clipcutter job (2 clips) | 3–7 min | Async after ack |
| n8n → Vercel callback delivery | < 5s | Final HTTP node |

The Vercel side polls the database every 5–6 seconds. As soon as your
callback writes the result, the UI surfaces it within the next poll.

## Test plan for the n8n side

After both webhooks + the callback wiring are in place, run these
through n8n directly (curl or n8n test interface) to validate
contracts without the Vercel UI:

**Transcription happy path:**
```bash
curl -X POST '<your transcription webhook URL>' \
  -H 'Content-Type: application/json' \
  -d '{
    "job_id":         "test-1",
    "session_id":     "<a real session_id from the sms_srp_generation table>",
    "source_url":     "https://www.youtube.com/watch?v=dQw4w9WgXcQ",
    "source_type":    "youtube",
    "callback_url":   "https://webhook.site/<your-debug-receiver>",
    "callback_secret": "test"
  }'
```

The callback receiver should fire within ~5 minutes with the
transcript JSON shape above.

**Clipcutter happy path:**
```bash
curl -X POST '<your clipcutter webhook URL>' \
  -H 'Content-Type: application/json' \
  -d '{
    "job_id":     "test-1",
    "session_id": "<real session_id>",
    "source_url": "https://example.com/sermon.mp4",
    "clips": [
      { "clip_id": "1", "clip_name": "Test", "in_point_ms": 0, "out_point_ms": 30000, "duration_ms": 30000, "quote": "test", "category": "hook" }
    ],
    "callback_url":   "https://webhook.site/<your-debug-receiver>",
    "callback_secret": "test"
  }'
```

Once both work via curl, the Vercel UI flow ("Transcribe" + "Cut clips"
buttons) is just the same thing with auth + UI polling.
